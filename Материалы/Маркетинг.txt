
1. Позиционирование (самое важное)
Это не «ещё один ORM», а "Convention-first auto-relations for EF Core via compile-time DSL"
Или иначе “Опиши связи интерфейсами — получи полноценную EF-модель без бойлерплейта”

2. Новизна

2.1. Сейчас в EF
- писать FK + navigation руками
- дублирование
- Fluent API шум

2.2. Здесь
class Order : IParent<User>, IParent<Tenant>
или
class Order : IParent<User, Tenant>
или
class Order : IParent<(User, Guid), Tenant>

3. MVP (без этого не публиковать)
3.1. Generator (incremental, обязательно)
Не обычный ISourceGenerator, а IIncrementalGenerator
иначе:
- медленный билд
- минус к adoption

3.2.  Диагностика (killer feature), добавить compile-time ошибки, примеры:
- CORM001: Duplicate relation 'User' in Order
- CORM002: Type 'User' must implement IEntity
- CORM003: Navigation already exists

3.3. Zero-config UX
dotnet add package CORM.AutoRelations
без:
- регистрации сервисов
- лишних вызовов

3.4. Работает с миграциями EF
Проверить dotnet ef migrations add Init

4. DX (developer experience)
4.1. IntelliSense должен видеть свойства:
order.User // должен быть виден

4.2. Навигация по коду
Ctrl+Click → переход в .g.cs

4.3. Ноль магии runtime
Никаких builder.ApplySomethingMagic()

6. README (это 70% успеха)
6.1. Hook (первые 5 секунд)

8. Что убивает такие проекты
8.1. Слишком сложная идея. Если нужно объяснять 10 минут — никто не будет использовать.

8.3. Баги в edge-case
- many-to-many
- composite keys
